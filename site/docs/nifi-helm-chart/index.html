NiFi Helm chart

Choose the Helm chart path that matches your Apache NiFi 2 deployment.

NiFi-Fabric keeps the standard Kubernetes install story centered on Helm. That makes the install, upgrade, and GitOps surface easier to understand than a controller-first design.

Chart surfaces

Use the platform chart for the standard customer path

The main entry point is charts/nifi-platform. It wraps the standard managed deployment shape and is the best fit when you want one release that installs the operator, the NiFi workload, and the opinionated defaults that NiFi-Fabric documents.

The reusable charts/nifi chart stays available for standalone-first or more explicit app-level installs. Together, the two charts keep the install surface boring and explainable while still covering the common platform shapes.

  • Use charts/nifi-platform for the recommended managed deployment.
  • Use charts/nifi when you need the lower-level reusable app chart.
  • Keep configuration in values files for GitOps-friendly rollouts.
  • Prefer the cert-manager-first install profile unless you need explicit secret ownership.

Helm-first means cleaner GitOps review

The standard install path is easy to audit because the resource ownership model is explicit. Values stay in the chart surface, certificates stay with cert-manager, and the operator only handles the lifecycle work that needs runtime awareness. 

helm upgrade --install nifi charts/nifi-platform \
  --namespace nifi \
  --create-namespace \
  --set global.nifiFabric.installProfile=quickstart-cert-manager \
  --set nifi.tls.certManager.issuerRef.name=nifi-ca
