Apache NiFi Kubernetes

Apache NiFi on Kubernetes without a heavyweight operator stack.

NiFi-Fabric is built for teams that want Apache NiFi 2 on Kubernetes with Helm-first installs, cert-manager-first TLS, and a thin controller that focuses on rollout and lifecycle safety.

What this page covers

Why NiFi-Fabric fits Apache NiFi on Kubernetes

Apache NiFi on Kubernetes usually goes wrong when the platform model becomes too controller-heavy or too magic to explain. The NiFi-Fabric approach keeps ordinary Kubernetes resources in Helm and reserves controller logic for the safety-sensitive work that Helm cannot coordinate on its own.

That means the standard install path stays explicit, GitOps-ready, and easy to reason about. Teams can see where StatefulSets, Services, Certificates, and Secrets come from, while still getting support for restart-aware rollout behavior and guarded lifecycle actions.

  • Helm owns the standard Kubernetes resource model.
  • cert-manager owns workload certificates on the default path.
  • The controller focuses on rollout, restore, and restart safety.
  • AKS is the primary supported target, with OpenShift-friendly overlays.

Recommended shape

  • Use charts/nifi-platform for the standard managed install.
  • Start with internal ClusterIP service exposure.
  • Back NiFi repositories with PVCs from a suitable StorageClass.
  • Layer ingress, load balancers, or service mesh only when needed.

How the Kubernetes model stays simple

NiFi-Fabric is intentionally not a broad NiFi controller stack. It keeps CRDs minimal, prefers controller-runtime conventions, and leans on native Kubernetes plus Helm wherever possible. That makes the platform easier to adopt for teams who already manage GitOps pipelines, cert-manager, and standard cluster networking. 

helm upgrade --install nifi charts/nifi-platform \
  --namespace nifi \
  --create-namespace \
  --set global.nifiFabric.installProfile=quickstart-cert-manager \
  --set nifi.tls.certManager.issuerRef.name=nifi-ca
