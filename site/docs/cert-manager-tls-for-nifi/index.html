cert-manager TLS for NiFi

Keep Apache NiFi TLS certificate management explicit and restart-safe.

NiFi-Fabric is TLS-first. The standard path uses cert-manager for workload certificates so the Kubernetes security model stays clear and the rollout logic can react safely to certificate changes.

Default TLS model

How cert-manager fits the NiFi-Fabric install path

On the standard path, Helm renders the Certificate object and cert-manager creates the final workload TLS Secret. NiFi-Fabric then keeps lifecycle safety front and center by treating certificate rotation and restart handling as first-class runtime concerns instead of hiding them behind a large controller surface.

  • Install cert-manager before the NiFi-Fabric release.
  • Create or choose an Issuer or ClusterIssuer for NiFi certificates.
  • Use the quickstart cert-manager profile for the fastest supported start.
  • Move to explicit secret ownership only when your platform needs it.

trust-manager remains optional. It is useful when you need a shared CA bundle for outbound trust or metrics scraping, but it does not replace cert-manager and it does not move TLS orchestration into the operator.

Standard cert-manager install command

The supported quickstart keeps the bootstrap simple while still pointing the final workload certificate at your chosen issuer. 

helm upgrade --install nifi charts/nifi-platform \
  --namespace nifi \
  --create-namespace \
  --set global.nifiFabric.installProfile=quickstart-cert-manager \
  --set nifi.tls.certManager.issuerRef.name=nifi-ca
